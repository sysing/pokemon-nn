# v0.3 — 4-Channel Input (RGB + Grayscale)

**Date**: 2026-06-14
**Dataset**: 855 images, 10 types
**Device**: M4 MacBook Air (MPS)

## Changes from v0.2

- Replaced `RandomGrayscale` with **stacked 4-channel input** — every image is `[R, G, B, Gray]`
- First conv layer: `Conv2d(3→32)` → `Conv2d(4→32)`
- Grayscale channel computed via **per-pixel** luminance formula:
  ```
  Gray[x,y] = 0.299·R[x,y] + 0.587·G[x,y] + 0.114·B[x,y]
  ```
  (ITU-R BT.601 standard weights)
- Input tensor shape: `[batch, 4, 224, 224]` — 200,704 values per image (50,176 pixels × 4 channels)

## Hypothesis

Instead of *randomly removing* color (v0.2), give the model **both** RGB and grayscale simultaneously. The model learns a soft gating mechanism — attend to RGB when color is distinctive (Fire=red, Water=blue), attend to grayscale when color is ambiguous (Metal vs Colorless, both silver).

## Insights

- **Hypothesis confirmed beyond expectations**: 4-channel approach achieves best of both worlds — preserves color-based classification while achieving 80% Metal accuracy (+45pp over baseline, +20pp over RandomGrayscale).
- **No trade-off**: Unlike v0.2 where overall accuracy dropped 5.2pp, v0.3 achieves the highest overall accuracy of all versions (94.7%).
- **First conv layer acts as a learned gate**: The `Conv2d(4→32)` layer learns separate filter weights for RGB vs grayscale channels. Downstream layers then learn when to attend to which pathway — effectively implementing a learned soft-gating mechanism without needing a multi-branch architecture.
- **Fighting still slightly degraded** (82.4% vs 100% in v0.1) — likely due to Fighting's brown/tan palette being similar to other earthy colors in grayscale, confusing the model when it weighs the grayscale channel too heavily.
- **Approach B (multi-branch) may still help**: With explicit RGB/Gray branches, we could regularize each branch's contribution independently, potentially recovering the lost Fighting accuracy.
